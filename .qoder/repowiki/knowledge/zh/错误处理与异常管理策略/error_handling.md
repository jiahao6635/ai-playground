该仓库采用**基于 Python 原生机制的防御性编程**策略，未引入专门的错误处理框架或全局中间件。核心模式包括：

1. **配置校验与快速失败 (Fail-Fast)**：在 `common/config.py` 中，通过显式检查环境变量并在缺失时抛出 `ValueError`，确保应用在启动阶段即暴露配置问题，避免运行时出现难以追踪的连接错误。

2. **工具层的容错封装 (Fault Tolerance)**：在 `05-tool-assistant/tools.py` 等模块中，针对外部依赖（如知识库检索）或高风险操作（如数学表达式计算），广泛使用 `try...except Exception` 捕获异常。关键约定是**不向上层抛出原始异常**，而是将其转化为包含错误信息的字符串返回给 LLM。这种设计允许 Agent 感知到工具执行失败并尝试自我修正或向用户反馈，体现了 AI 应用特有的“错误即信息”的处理逻辑。

3. **交互式 CLI 的全局兜底**：在各个项目的 `main.py` 交互循环中，统一使用 `try...except Exception as e` 包裹核心调用逻辑。这确保了单次请求的失败（如网络波动、模型超时）不会导致整个 CLI 进程崩溃，提升了学习沙盒的鲁棒性。

4. **工作流中的安全约束**：在 `09-code-approval/graph.py` 中，通过静态关键词扫描（如禁止 `import os`）和受限的 `exec` 环境来预防代码执行风险，并在执行节点内部捕获运行时异常，将错误结果作为状态的一部分持久化。

**开发规范建议**：
- 在编写新工具时，必须捕获所有潜在异常并返回描述性字符串。
- 配置加载逻辑应遵循“尽早校验”原则。
- 避免在业务逻辑中吞没异常，除非已将其转化为对 LLM 有意义的上下文信息。
